# Think - Structured Planning Mode

You invoked `/think`. This means you want a thorough analysis before writing code. Let me load the planning tools and think through this.

**Topic**: $ARGUMENTS

## Process

1. **Understand** - Restate the problem/goal to confirm alignment
2. **Explore** - Load `brainstorming` skill to explore options, constraints, risks
3. **Plan** - Load `writing-plans` skill to produce a structured plan
4. **Deliver** - Present the plan with clear next steps

## What a Good Plan Includes

- Problem statement (what are we solving, and why?)
- Success criteria (how will we know it's done?)
- Approach (architecture, patterns, key decisions)
- Steps (ordered, concrete, testable)
- Files touched (which files and why)
- Risks (what could go wrong, and how we'll handle it)
- Open questions (what we still need to decide)

## Constraints

- No code changes during planning (read-only analysis)
- Consider the 8 principles before recommending
- If the problem doesn't need solving, say so
- If it can be solved with a simpler approach, recommend that instead
