# CWE Top 25 Most Dangerous Software Weaknesses (2025)

Source: https://cwe.mitre.org/top25/archive/2025/2025_cwe_top25.html

| Rank | CWE | Name | Prevention |
|------|-----|------|------------|
| 1 | CWE-79 | Cross-site Scripting (XSS) - Untrusted input rendered in web pages without encoding | Context-aware output encoding; use CSP; avoid innerHTML |
| 2 | CWE-89 | SQL Injection - User input included in SQL queries without parameterization | Use parameterized queries/prepared statements exclusively |
| 3 | CWE-352 | Cross-Site Request Forgery (CSRF) - Forged requests exploit authenticated sessions | Anti-CSRF tokens; SameSite cookies; verify Origin header |
| 4 | CWE-862 | Missing Authorization - No access control check before resource access | Enforce authorization on every request at the controller/middleware layer |
| 5 | CWE-787 | Out-of-bounds Write - Writing past buffer boundaries corrupts memory | Bounds checking; use memory-safe languages (Rust, Go); avoid raw pointer arithmetic |
| 6 | CWE-22 | Path Traversal - User input used to construct file paths without sanitization | Canonicalize paths; validate against allowlist; chroot/jail file access |
| 7 | CWE-416 | Use After Free - Accessing memory after deallocation | Use smart pointers; memory-safe languages; static analysis tools |
| 8 | CWE-125 | Out-of-bounds Read - Reading beyond buffer limits leaks data | Bounds checking; safe string/buffer APIs; fuzzing |
| 9 | CWE-78 | OS Command Injection - User input passed to shell commands | Avoid shell execution; use parameterized APIs; strict input validation |
| 10 | CWE-94 | Code Injection - User input evaluated as code at runtime | Never eval() user input; use sandboxes; strict input validation |
| 11 | CWE-120 | Classic Buffer Overflow - Copying input without checking buffer size | Use safe copy functions (strncpy, snprintf); bounds checking |
| 12 | CWE-434 | Unrestricted File Upload - Uploaded files executed on server | Validate file type server-side; store outside webroot; rename files |
| 13 | CWE-476 | NULL Pointer Dereference - Accessing null reference crashes application | Null checks before dereference; use Option/Optional types; static analysis |
| 14 | CWE-121 | Stack-based Buffer Overflow - Stack buffer overwritten by oversized input | Stack canaries; ASLR; avoid alloca/VLAs; use safe string functions |
| 15 | CWE-502 | Deserialization of Untrusted Data - Malicious serialized objects trigger code execution | Never deserialize untrusted data; use allowlists for classes; prefer JSON over native serialization |
| 16 | CWE-122 | Heap-based Buffer Overflow - Heap buffer overwritten enables arbitrary code execution | Bounds checking; safe allocators; memory-safe languages |
| 17 | CWE-863 | Incorrect Authorization - Authorization check exists but is flawed | Test authorization logic with negative cases; principle of least privilege |
| 18 | CWE-20 | Improper Input Validation - Missing or incomplete input validation | Validate all input against strict schemas; allowlist over denylist |
| 19 | CWE-284 | Improper Access Control - General access control failures | Centralized access control framework; deny by default |
| 20 | CWE-200 | Information Exposure - Sensitive data leaked in responses or logs | Minimize data in error messages; review API responses; mask sensitive fields |
| 21 | CWE-306 | Missing Authentication for Critical Function - No auth on sensitive endpoints | Require authentication for all non-public endpoints; audit exposure |
| 22 | CWE-918 | Server-Side Request Forgery (SSRF) - Server makes requests to attacker-controlled destinations | Validate/allowlist URLs; block private IP ranges; disable redirects |
| 23 | CWE-77 | Command Injection - Special elements in commands not neutralized | Use library APIs instead of shell; parameterize commands |
| 24 | CWE-639 | Authorization Bypass via User-Controlled Key - Attacker modifies key to access others' data | Map to server-side session; never trust client-supplied keys for authz |
| 25 | CWE-770 | Resource Allocation Without Limits - Unbounded allocation enables DoS | Set resource limits (memory, connections, CPU); rate limiting |

## Key Themes

**Memory Safety** (Rank 5, 7, 8, 11, 14, 16): Use memory-safe languages or enable compiler protections (ASLR, stack canaries, CFI).

**Injection** (Rank 1, 2, 9, 10, 23): Never concatenate untrusted input into interpreted strings. Parameterize everything.

**Access Control** (Rank 4, 17, 19, 21, 24): Centralize authz, deny by default, test negative cases.
